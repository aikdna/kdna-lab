Force-pushing **rewritten history** (e.g., an interactive rebase or commit amend) to a **shared `main` branch** is almost always a bad idea. Here’s why, and what to do instead:

## Why it’s dangerous

- **Breaks other developers’ local clones** – If anyone has pulled the old history, their next `git pull` will fail with conflicts or create a mess.
- **Loss of commits** – If someone’s work was based on a commit you rewrote, that work may appear orphaned or lost.
- **Team confusion** – Git relies on a shared, stable history. Rewriting it undermines trust and makes collaboration harder.

## Better alternatives

1. **Use feature branches** – Do all rewriting (squashing, reordering, amending) on a private or short-lived feature branch. Force-push to that branch freely, then merge cleanly into `main`.
2. **Coordinate with the team** – If rewriting `main` is absolutely necessary (e.g., to remove a leaked secret), communicate beforehand, have everyone stash or commit their work, and agree on a window of time.
3. **Use `--force-with-lease`** – Safer than `--force` because it checks that your remote-tracking branch hasn’t been updated by someone else. Still, only use on non-shared branches.
4. **Prefer revert over rewrite** – To undo changes in `main`, use `git revert` instead of rewriting history. This preserves the commit history and is safe for everyone.

## Bottom line

**Never force-push rewritten history to a shared branch like `main`** unless you have explicit, unanimous team consent and a clear plan to handle the fallout. In most teams, this is a violation of the agreed-upon workflow.
