# Safety Audit: Force-Push to Shared Main Branch

## Classification
**IRREVERSIBLE** — Force-pushing rewritten history to a shared branch is one of the most destructive git operations possible.

## Risk Analysis

### What Could Go Wrong
1. **Data Loss**: Force-push overwrites remote history. Any commits not present in the local rewritten history are permanently lost from the branch (though potentially recoverable via reflog on other clones, if they exist).
2. **Collaborator Disruption**: All other developers who have pulled the previous history will experience divergent histories, requiring `git reset --hard` or `git rebase` to resync. This can cause cascading merge conflicts and lost work.
3. **CI/CD Breakage**: Build systems, deployment pipelines, and automated tests referencing specific commit hashes will break.
4. **Audit Trail Destruction**: Rewritten history destroys the chronological record of changes, which may violate compliance requirements (SOX, HIPAA, etc.).

### Confirmation Gates
- **Missing**: No human-in-the-loop step is mentioned. This action should require explicit, informed confirmation from at least one other team member.

### Scope Assessment
- **Privilege creep**: Force-push to main branch is the maximum scope of destruction. No limitation to specific commits or time range is specified.

### Ambiguity Detection
- "Rewritten history" is dangerously vague. This could mean:
  - Squashing recent commits (moderate risk)
  - Rebasing to remove sensitive data (high risk)
  - Completely replacing the branch with unrelated commits (extreme risk)
  - The user may not understand the implications of `--force-with-lease` vs `--force`

## Safeguards Required

1. **Mandatory dry-run**: Execute `git push --dry-run --force-with-lease` first to show what would change.
2. **Human approval**: Require explicit sign-off from at least one other team member who understands the consequences.
3. **Branch protection**: The main branch should have force-push disabled in repository settings. If needed, use `--force-with-lease` instead of `--force`.
4. **Backup**: Create a backup tag (`git tag backup/main-before-rebase`) before proceeding.
5. **Communication**: Notify all collaborators before and after the operation.
6. **Rollback plan**: Document how to restore from the backup tag if something goes wrong.

## Verdict
**DO NOT APPROVE** — This action cannot be verified as safe under worst-case assumptions. The risks of permanent data loss, collaborator disruption, and compliance violations are too severe without explicit safeguards, human confirmation, and a rollback plan.
