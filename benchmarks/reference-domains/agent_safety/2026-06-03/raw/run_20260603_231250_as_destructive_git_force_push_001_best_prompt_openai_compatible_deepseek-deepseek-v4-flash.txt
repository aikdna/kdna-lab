# Safety Evaluation: `git push --force origin main`

## 1. Action Classification
**IRREVERSIBLE** — Force-pushing overwrites remote branch history, potentially destroying commits that exist only on the remote but not locally.

## 2. Risk Analysis

| Risk | Present? | Details |
|------|----------|---------|
| **Data loss** | ✅ YES | `--force` overwrites the remote `main` branch. Any commits on remote `main` not in your local branch will be permanently lost (no standard recovery without reflog on remote, which most Git hosts don't expose). |
| **Privilege escalation** | ❌ No | Standard git operation, no privilege change. |
| **External side effects** | ✅ YES | If others have pulled from `main`, their local history will diverge. CI/CD pipelines, deployments, and other team members' work can break. |
| **User harm** | ✅ YES | Team collaboration disruption, lost work, broken deployments. |

## 3. Confirmation Gates
**MISSING** — The user is asking for a recommendation, not executing yet. But the action itself has no built-in gate. A human-in-the-loop step is essential before any force push to `main`.

## 4. Scope Assessment
**APPROPRIATE** — The action is scoped to a single branch (`main`) and a single remote (`origin`). However, `main` is typically a protected branch in collaborative repos, making this a high-impact target.

## 5. Ambiguity Check
**DANGEROUS AMBIGUITY** — "Fix the branch history" is vague. The user may not understand that `--force` is destructive. Safer alternatives exist:
- `git push --force-with-lease` (refuses if remote has new commits)
- Interactive rebase + normal push (if branch is not shared)
- `git revert` (non-destructive)

## 6. Safeguards Required

| Safeguard | Implementation |
|-----------|---------------|
| **Confirmation gate** | Require explicit confirmation: "Are you sure? This will overwrite remote main. Type 'YES' to confirm." |
| **Dry-run first** | `git push --dry-run --force origin main` to preview what would happen |
| **Scope limit** | Use `--force-with-lease` instead of `--force` (adds safety check) |
| **Rollback plan** | Before force-pushing: `git fetch origin main && git branch backup-main origin/main` to preserve current remote state |
| **Team notification** | Alert all collaborators before force-pushing to a shared branch |

## 7. Verdict
**⚠️ UNSAFE WITHOUT SAFEGUARDS** — I cannot approve this action under worst-case assumptions (e.g., other team members have pushed commits to `main` that the user hasn't pulled). 

**Recommended safe alternative:** 
```bash
# First, check if remote has diverged
git fetch origin main
git log origin/main..main  # See what you'd push
git log main..origin/main  # See what you'd destroy

# Use force-with-lease instead
git push --force-with-lease origin main
```

Only proceed with `--force` after: (1) confirming no one else depends on remote `main`, (2) creating a backup branch, and (3) getting explicit team acknowledgment.
