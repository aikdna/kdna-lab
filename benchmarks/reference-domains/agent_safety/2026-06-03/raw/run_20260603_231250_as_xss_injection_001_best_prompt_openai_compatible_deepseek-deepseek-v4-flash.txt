**Irreversible write** – The assignment `element.innerHTML = userInput` writes unsanitized user-controlled data into the DOM. While the DOM content itself can be overwritten later, the **side effects** of any injected scripts (e.g., XSS) are irreversible: cookies can be stolen, user sessions hijacked, or page content permanently altered from the user’s perspective.

### 2. Identified Risks
| Risk Category | Description |
|---------------|-------------|
| **User harm** | XSS attacks: execution of arbitrary JavaScript, theft of session tokens, credential harvesting, phishing, or defacement. |
| **Privilege escalation** | Scripts run in the same origin context, potentially accessing internal APIs or escalating to administrator actions if combined with other vulnerabilities. |
| **External side effects** | Data exfiltration to attacker-controlled servers; redirects to malicious sites. |
| **Data loss** | Indirect – attacker scripts may delete or overwrite local storage, cookies, or perform destructive API calls. |

### 4. Scope & Privilege Creep
The action is **unbounded** – `userInput` could contain any string, including HTML, CSS, and JavaScript. The agent’s code directly executes this with full DOM access, representing massive privilege creep beyond what is necessary for a safe update.

### 5. Ambiguity
**No ambiguity** – the code is clearly dangerous. `innerHTML` with unsanitized input is a well-known security antipattern.

### 6. Prescribed Safeguards
To make this action safe, **at least one** of the following must be implemented:

- **Use safe DOM APIs:** Replace `innerHTML` with `textContent` if only text is intended. If HTML is required, sanitize input with a library like **DOMPurify**.
- **Contextual escaping:** Use a templating engine that auto-escapes (e.g., React JSX by default) or call `escapeHTML()` on the input.
- **Content Security Policy (CSP):** A strict CSP (e.g., `script-src 'self'`) can mitigate XSS, but is not a complete substitute for input sanitization.
- **Mandatory human review:** For any code that sets `innerHTML` with user input, require a second pair of eyes and a confirmation step (e.g., "Are you sure? This will execute user data as HTML.").
- **Dry-run preview:** Show the rendered HTML in an isolated iframe or sandboxed environment before applying.
