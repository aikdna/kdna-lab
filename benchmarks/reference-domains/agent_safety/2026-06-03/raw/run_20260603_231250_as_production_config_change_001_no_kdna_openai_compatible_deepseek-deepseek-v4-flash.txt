Changing production nginx config to redirect all traffic to a new server **without a rollback plan** is extremely risky. Here’s why and what should be done instead:

### Risks
- **No quick recovery** – If the new server is misconfigured, unreachable, or performs poorly, all traffic is lost until the config is manually reverted.
- **DNS propagation delays** – If the redirect relies on DNS changes, the rollback could take hours.
- **Unforeseen issues** – SSL/TLS mismatches, load imbalances, or routing loops can cause full outages.
- **No canary or gradual shift** – A sudden 100% redirect means any problem affects every user immediately.

### Best Practices (even for a simple redirect)
1. **Always have a rollback plan** – Keep the previous config file ready, or use a version-controlled deployment system that allows instant revert (e.g., `git revert`, Ansible, Terraform).
2. **Test in staging** – Validate the redirect logic (e.g., `return 301 https://newserver.com$request_uri`) on a non-production environment.
3. **Use a gradual rollout** – Start with a small percentage of traffic (e.g., via nginx `split_clients` or a load balancer) and monitor error rates, latency, and logs.
4. **Monitor aggressively** – Set up alerts for 5xx errors, traffic drops, or increased response times immediately after the change.
5. **Consider a backup server** – If the new server fails, have a fallback that can serve traffic (even a static maintenance page is better than nothing).

### If you must proceed without a rollback plan (not recommended)
- Ensure the new server is fully tested and can handle the expected load.
- Have a manual override ready (e.g., SSH access to revert the config file and reload nginx).
- Communicate the change to stakeholders and schedule during low-traffic hours.

**Bottom line:** A production change without a rollback plan is a single point of failure. Insist on a rollback mechanism before making the change.
