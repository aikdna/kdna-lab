### 1. 识别实际要求与隐含假设

**明确要求：**  
- 分析给定“代码”的性能瓶颈。  
- 将这段代码翻译成Python。  
- 写一个README文档解释如何使用（翻译后的Python代码）。  

**隐含假设：**  
- 存在一段具体的“代码”（未提供）。  
- 这段代码不是Python，需要被翻译。  
- 性能瓶颈分析是指对原始代码的分析，而不是对翻译后的Python代码的分析（“它”指代的是原始代码）。  
- README文档的内容是基于翻译后的Python代码，并且需要包含使用说明。  
- 用户期望AI同时完成这三项任务，但未说明优先级或输出顺序。

### 2. 任务歧义检测

| 歧义类型 | 具体表现 |
|---------|----------|
| 多目标混合 | 三个任务（分析瓶颈、翻译、写文档）被合并为一个指令，但彼此依赖关系不明确。例如：分析瓶颈是否需要基于原始代码？翻译后是否还要重复分析？ |
| 成功标准不清晰 | “性能瓶颈”没有定义衡量的指标（时间复杂度？空间？I/O？）；“翻译”是否要求逐行等价？是否允许重构优化？“README”应包含哪些章节（安装？示例？API？）没有说明。 |
| 范围模糊 | “这段代码”缺乏上下文——是伪代码？其他语言？多长？是否包含外部依赖？ |

### 3. 缺失上下文

- **原始代码缺失**：最根本的缺失。没有代码就无法执行任何任务。  
- **性能瓶颈分析的目标语言**：是分析原始代码还是翻译后的Python？  
- **翻译的约束**：是否需要保持原逻辑的精确性？是否允许引入Pythonic特性（如列表推导）？  
- **README的受众**：是面向最终用户还是开发者？  
- **额外信息**：代码的运行环境（如单线程/多线程、数据规模等）对瓶颈分析至关重要。

### 4. 输出格式期望

- 未指定任何格式。AI需要自行决定输出是纯文本、分段、列表、代码块还是Markdown。  
- 对于“翻译成Python”，期望输出代码块。  
- 对于“README”，期望结构化文档（但结构未定义）。  
- 对于“分析瓶颈”，期望分析报告（但无模板）。

### 5. 风险措辞分析

- **“把它翻译成Python”** — 代词“它”指代不明，易导致AI理解偏差（可能将瓶颈分析结果“翻译”成Python？）。  
- **“同时写一个README”** — 使用“同时”暗示并行执行，但AI的回复是顺序的，可能遗漏或混排输出。  
- **隐含的安全风险**：如果原始代码包含密码、API Key等敏感信息，要求直接翻译并写文档可能导致泄露。但此风险属于使用者问题，而非prompt本身的设计缺陷。

### 6. 具体重写建议（修复每个缺陷）

**修复后的Prompt：**

```
请针对以下提供的代码（附在消息中）依次完成三项任务：  
1. **性能瓶颈分析**：  
   - 分析该代码在执行效率（时间/空间复杂度）、资源消耗（内存/IO）方面的主要瓶颈。  
   - 假设代码运行在单线程环境下，输入数据规模为[-在此描述典型规模-]。  
   - 以要点形式输出分析结果。  

2. **翻译为Python**：  
   - 将原始代码逐行翻译为等效的Python代码，保持逻辑完全一致，不进行优化（除非优化不改变行为）。  
   - 使用Python 3.8+的语法，保留注释。  
   - 输出时使用```python代码块。  

3. **编写README文档**：  
   - 基于翻译后的Python代码，编写一份面向开发者的README文档，内容需包括：  
     - 项目简介  
     - 环境要求（Python版本、依赖库）  
     - 安装步骤  
     - 使用示例（含输入输出样例）  
     - 参数说明（如有函数参数）  
   - 使用Markdown格式输出。  

请将以上三项任务的结果按顺序分隔输出，第一条输出为“### 性能瓶颈分析”，第二条为“### Python代码”，第三条为“### README文档”。  
```

**修复说明：**  
- **消除歧义**：明确“它”指原始代码；分开三个任务并指定顺序。  
- **提供上下文**：要求用户先提供代码；假设运行环境和数据规模。  
- **明确成功标准**：瓶颈分析指定了效率与资源；翻译要求“逐行等效”；README指定了章节结构。  
- **格式显式化**：输出顺序、代码块、Markdown、标题分隔。  
- **避免并行词**：用“依次”代替“同时”。  
- **风险控制**：无敏感信息风险，但若后续用户提供敏感代码，此提示不会要求公开。
