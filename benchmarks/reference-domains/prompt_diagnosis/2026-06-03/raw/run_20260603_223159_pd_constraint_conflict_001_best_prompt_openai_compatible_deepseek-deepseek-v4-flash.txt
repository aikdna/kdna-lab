## 诊断分析

### 1. 实际意图与隐含假设

- **显式意图**：要求生成一篇技术报告，满足三个属性——10000字、详细、简洁、5分钟内可读完。
- **隐含假设**：用户可能希望报告既全面又精炼，但混淆了“字数”与“阅读时长”的可量化的矛盾关系。正常中文阅读速度约为200–300字/分钟，5分钟最多阅读1500字，而要求10000字是前者的6–7倍，物理上不可能同时满足。

### 2. 任务模糊性

- **多目标冲突**：同时要求“10000字”和“简洁”是语义上的直接对抗；要求“详细”又与“5分钟内读完”矛盾。
- **成功标准不明**：模型无法判断应该优先满足字数、内容密度还是阅读时长。几乎没有一种输出能同时满足所有条件。

### 3. 缺失上下文

- **报告主题**：未指定任何技术领域或问题（如“关于XX系统的性能分析”），导致模型无法定向收集知识。
- **目标读者**：未说明是面向专家、管理者还是普通读者，影响语言复杂度。
- **格式与结构**：是否需要章节、图表、参考文献？未提供。
- **风格限制**：“简洁”的具体含义（无废话？短句？高度概括？）未定义。

### 4. 输出格式期望

- **隐含格式**：纯文本报告，但未明确要求Markdown、Word、PDF等。
- **可能的误解**：模型可能理解为“写一个10000字的文本字符串”，输出极长但内容空洞并重复，或放弃执行并提示矛盾。

### 5. 风险措辞

- **否定/矛盾陷阱**：“要简洁”与“10000字”形成经典悖论。模型可能尝试折中（如用极小的字体压缩篇幅），但这不是合理的技术输出。
- **过度宽松**：没有给出任何边界条件，模型可能自行解释“简洁”为“每行很短”，但依然无法解决字数总量问题。

### 6. 严重性评级

| 问题类别 | 严重程度 | 说明 |
|---------|---------|------|
| 任务逻辑矛盾 | **功能性** | 导致指令不可执行，模型要么拒绝，要么产出违反正规要求的无效内容。 |
| 上下文缺失 | **功能性** | 缺少主题和读者，报告内容无法聚焦。 |
| 输出格式模糊 | **外观性** | 可通过默认格式缓解，但最好明确。 |
| 风险措辞 | **功能性** | “简洁”与字数要求互斥，属于核心冲突。 |

### 7. 具体重写方案

**原Prompt（问题）**：
> 写一篇10000字的详细技术报告，但要简洁，5分钟内能读完。

**重写Prompt（修正版）**：
> 请撰写一篇关于【请在此处填入具体技术主题，例如“分布式数据库的CAP理论在现代云环境下的应用”】的技术报告。
> 要求：
> 1. 总字数控制在1200–1500字（确保正常阅读速度下5分钟内完成）。
> 2. 内容详实但不冗余：每个关键点用1–2句话精炼说明，避免展开无关背景。
> 3. 面向【目标读者，例如“有一定数据库基础的技术经理”】撰写，使用简洁、专业的语句。
> 4. 结构包含：摘要（2–3句）、问题背景、核心分析、结论与建议（可选）。
> 5. 输出格式：Markdown，标题和段落清晰。

**每项修正说明**：

| 修正内容 | 解决的问题 |
|---------|-----------|
| 明确指定技术主题（占位符） | 补充缺失的上下文，让模型有定向知识来源 |
| 将字数改为1200–1500字 | 消除与“5分钟内读完”的物理矛盾；保留“详细”的语义通过精炼内容实现 |
| 解释“简洁”为“不冗余”“精炼说明” | 消除语义模糊，将“简洁”转化为可操作规则 |
| 指定目标读者和语言风格 | 提供约束条件，避免报告过于学术或过于浅显 |
| 规定结构（摘要、背景、分析、结论） | 使输出格式可预期，避免无组织的长篇大论 |
| 要求Markdown格式 | 明确输出格式，方便后续使用 |

这样重写后，模型能明确知道：字数范围、内容密度标准、受众、结构、格式。成功率从近乎0提升到接近100%。
