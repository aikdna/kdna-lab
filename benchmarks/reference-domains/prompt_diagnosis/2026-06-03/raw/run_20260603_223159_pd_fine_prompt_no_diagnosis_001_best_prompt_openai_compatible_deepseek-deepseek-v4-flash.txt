## 诊断报告：用户提示分析

### 用户原始输入
```
"请以JSON格式输出以下三个产品的对比：价格、功能、用户评分。每个产品一段，包含优缺点。用中文。"
```

---

### 1. 实际请求 vs 隐含假设

| 项目 | 内容 |
|------|------|
| **显式请求** | 输出三个产品的对比，格式为JSON，每个产品用一段文字描述其价格、功能、用户评分，并包含优缺点。语言为中文。 |
| **隐含假设** | ① 用户已在上下文中指定了三个具体产品（但未给出）；② JSON 格式可以容纳“一段”文本；③ 用户评分、价格、功能数据已知或可凭空生成。 |

**问题**：核心依赖缺失——没有产品名称，导致无法执行。这是**功能性**缺陷（严重性：⭐️⭐️⭐️）。

---

### 2. 任务模糊性

- **混合目标**：同时要求“JSON格式”和“每个产品一段”。JSON本质上是结构化键值对，而“一段”暗示自然语言段落。二者语义冲突：若用JSON，每个产品的描述应拆分为字段；若用段落，则不应限定为JSON。
- **成功标准不明确**：对比的维度（价格、功能、用户评分）是否必须在JSON中作为独立字段？优缺点应该以何种粒度呈现？用户没有指定数据来源或参考标准（例如评分是10分制还是5星？价格是美元还是人民币？）。

**严重性**：功能性（⭐️⭐️⭐️），格式冲突可能导致输出既非有效JSON也非可读段落。

---

### 3. 缺失上下文

- **产品列表缺失**：必须补充三个具体产品名称或描述。
- **数据来源/背景**：价格、功能和用户评分是来自公开市场、用户调研还是模拟数据？是否需要附带单位或货币？
- **用户评分说明**：评分是平均值还是中位数？最近更新时间？是否有样本量要求？
- **优缺点标准**：优缺点基于什么维度（性价比、易用性、生态等）？是否由AI自行总结，还是需要引用来源？

**严重性**：功能性（⭐️⭐️⭐️），缺少这些信息会导致输出完全不可控或不符合用户预期。

---

### 4. 输出格式期望分析

用户要求“JSON格式”，但“每个产品一段”的描述暗示可能想要以下两种之一：

- **方案A（最可能）**：一个包含多个对象的JSON数组，每个对象含有字符串字段（如 `description`），该字段值为一段包含价格、功能、评分和优缺点的中文文本。  
- **方案B**：每个产品作为一个独立对象，其中价格、功能、评分、优点、缺点作为独立键值，但用户又要求“一段”，所以可能期望在某个字段中整合成段落。

**问题**：用户未明确是否要结构化字段（推荐）还是嵌入文本。缺少输出示例。

**严重性**：功能性（⭐️⭐️），可能导致AI按错误模式输出，但可通过合理推测解决。

---

### 5. 风险措辞分析

- **“优缺点”**：主观性强，没有限定判断标准，AI可能给出不准确或带偏见的评价（例如对特定品牌的倾向）。
- **“用户评分”**：没有指定评分来源或准确性，AI可能虚构数字，造成误导（安全风险，但通常为轻度）。
- **无否定陷阱或引导性问题**：此部分无严重风险。

**严重性**：安全-轻度（⭐️），可接受，但建议在重写中明确评分来源或要求注明“基于公开信息模拟”。

---

### 6. 具体重写方案

**原始缺陷**：
- 缺少产品列表
- 格式矛盾（JSON vs 段落）
- 评分/价格/优缺点缺乏规范

**重写建议**（替换整个提示）：

```text
请以JSON格式提供以下三个智能手机的对比：
- 产品A：iPhone 15 Pro Max
- 产品B：Samsung Galaxy S24 Ultra
- 产品C：Google Pixel 8 Pro

要求：
1. 输出一个JSON数组，每个元素对应一个产品。
2. 每个产品对象的字段如下（使用中文）：
   - "name": 产品名称
   - "price": 价格，以美元为单位，附上货币符号（如 "$1199"）
   - "features": 数组，列出3~5个主要功能（如"钛金属边框", "A17 Pro芯片"）
   - "user_rating": 数值，范围1.0~5.0，保留一位小数
   - "pros": 数组，列出该产品的优点（每条简短，不超过10字）
   - "cons": 数组，列出该产品的缺点
3. 价格和用户评分的数值请基于2024年~2025年的公开市场数据，你自己合理估计，但需在输出后注明“数据为模拟，仅供参考”。
4. 不要使用自然语言段落，所有信息通过字段表达。
5. 整个JSON应合法、无多余注释，使用中文填写字段内容。
```

**每个改动解决的问题**：

| 改动 | 对应缺陷 |
|------|----------|
| 明确三个具体产品 | 解决“缺少产品”的功能性缺陷 |
| 指定结构化字段（name, price等） | 解决格式矛盾，将“一段”替换为具体字段 |
| 给出数组/数值/字符串类型要求 | 消除输出格式歧义 |
| 要求说明数据来源 | 降低虚构数据的风险（安全-轻度） |
| 限定用户评分范围、价格单位 | 避免输出混乱 |
| 增加“基于公开市场数据”及免责声明 | 避免提供虚假信息 |
